    • Exercise #5, Is it possible to rewrite the previous example to be linear?

    I do not think it is possible for an arithmatic operation to be linear when trying to sort a list to find the n'th largest number. I'm certain that an algorithm that goes beyond trying to find the smallest or largest number will always end up being 0(n^2) or O(n log n). An algorithm that needs to compare numbers against every other number in a list cannot be linear in its timescale. This stands when n is any value, but as stated, I believe it is possible for it to be linear when n is looking for the smallest or largest number. It only needs to make one trip through the list when finding either the smallest or largest, picking up whichever it needs and dropping the previous value.

    • Exercise #6 Cont. Since chaining in a hashmap entails the use of a collection of multiple items at certain hashes, a deletion function would need to check each hash location for the possibility of a list. A specialized search function that could check each hash location and iterate through potential collections is necessary. If the search function found its intended key within a collection on the hashmap, the deletion of the element is simple. However, for open addressing, the case is quite different. Given that two items cannot reside within the same hash location, an item with a hash that would normally fit into a slot has to be slotted somewhere else. Deleting an item in a hashmap that has a similar hash id as another piece of data will cause a not-found error. A special data marker needs to be placed when deleting items in open addressed hash maps. These markers let the search function know that the item that was once here was deleted, but there are potentially other items with similar hash ids that may be elsewhere in the list.
